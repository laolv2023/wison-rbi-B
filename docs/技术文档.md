## 1. 系统概述

### 1.1 项目定位

Wison-RBI 是一个**浏览器隔离（Browser Isolation）系统**。它通过在 Chromium Compositor 层拦截绘制命令，使客户端浏览器不必执行任何远程 HTML/CSS/JavaScript，仅接收经过安全校验的 Skia 绘制命令。所有页面内容仅在服务端 Chromium 沙箱内执行，从根本上消除客户端被网页内容攻击的攻击面。

- **项目名称**: Wison-RBI
- **版本**: v1.6
- **代码仓库**: https://github.com/laolv2023/wison-rbi-B
- **许可证**: MIT
- **拦截层级**: Chromium Compositor 层（cc::DisplayItemList / PaintOp）

### 1.2 核心原理

```
传统网络浏览:
  客户端浏览器 ← HTTP → Web 服务器
  ↑ 客户端执行所有 HTML/CSS/JS，承载全部网页攻击面

Wison-RBI:
  客户端 (Chrome Extension MV3)
      ← WebSocket (binary frames) ←
  服务端 (Node.js + headless Chromium)
      ← HTTP → Web 服务器
  ↑ 客户端仅执行 Skia 绘制命令（不含 HTML/CSS/JS）
```

服务端运行 headless Chromium 实例，捕获 Compositor PaintOp，序列化为二进制帧，通过 WebSocket 推送给客户端。客户端接收帧后，通过 9 层 CommandValidator 校验，最终用 CanvasKit WASM 渲染到 `<canvas>`。

### 1.3 与 Cloudflare NVR 的关键区别

| 维度 | Cloudflare NVR | Wison-RBI |
|------|---------------|-----------|
| 拦截层级 | Skia API | Compositor 层 (cc::DisplayItemList / PaintOp) |
| Chromium 修改范围 | ~15 个文件 | 1-2 个文件 + 独立 RecordingCanvas |
| 客户端引擎 | 专有 WASM | CanvasKit WASM（开源标准 Skia） |
| 增量更新 | 未公开 | R-tree 空间索引驱动的脏区域 |
| 可审计性 | 闭源 | 全链路开源（MIT） |

## 3. 架构设计

### 3.1 系统整体架构

```

```

### 3.2 数据流

#### 下行流（服务端 → 客户端）：绘制帧

```
Chromium Compositor 产出 PaintOp List
  → CommandBuffer 序列化（C++ 层 / JS 层）
  → FrameAssembler 组帧
  → CRC32 完整性校验码计算
  → gzip 压缩（三层 zip bomb 防护）
  → WebSocket 二进制帧发送
  → 客户端解压 + CRC 校验
  → CommandValidator 9 层白名单校验
  → CanvasKit SkSurface 渲染
```

#### 上行流（客户端 → 服务端）：用户输入

```
用户物理输入（鼠标/键盘/滚轮）
  → 客户端 HID 事件捕获
  → WebSocket JSON 消息（不含页面语义）
  → InputProxy 坐标转换（canvas px → CSS px，方案 B scroll 锚定）
  → CDP Input.dispatchMouseEvent / Input.dispatchKeyEvent
  → Chromium 页面响应
```

### 3.3 核心设计原则

1. **Compositor 层拦截** — 在 Chromium 渲染管线的 Compositor 层记录 PaintOp，而非 Skia API 层。这大幅减少 Chromium 源码修改量（仅 1-2 个文件 + 独立 RecordingCanvas 子类）。
2. **纵深防御** — 服务端沙箱 + TLS 加密 + 客户端命令白名单。攻击者即使完全控制服务端，客户端仍然安全（不变量 1 + CommandValidator 9 层校验）。
3. **最小信息暴露** — 客户端仅发送原始 HID 事件，不传输页面语义；服务端仅发送 Skia 绘制命令，不传输 HTML/CSS/JS。
4. **Fail-Closed** — 任何校验失败 → 丢弃整帧，不尝试部分渲染或修复。CRC 失败、Opcode 非法、子结构超限均触发整帧丢弃。
5. **单一事实来源** — 协议常量在 `cpp/garnet_config.h` / `cpp/frame_constants.h` / `server/config.js` / `client/protocol.js` 四端严格一致，跨语言边界审计。

## 4. 通信协议规范

### 4.1 帧格式（二进制）

```
┌──────────┬──────────────────────┬────────────┐
│  30B     │       N bytes        │    4B      │
│  Header  │   CommandStream      │   CRC32    │
└──────────┴──────────────────────┴────────────┘
```

- **CRC32 覆盖范围**: Header (30B) + CommandStream (N bytes)
- **CRC32 多项式**: `0xEDB88320`（IEEE 802.3，与 zlib 兼容）
- **字节序**: Little Endian（除字体 Magic 字节白名单外，全协议统一小端）

### 4.2 帧头结构（30 字节，Little Endian）

> 验证来源: `cpp/frame_constants.h`

| 偏移 | 大小 | 字段 | 类型 | 说明 |
|------|------|------|------|------|
| 0 | 1B | version | uint8 | 协议版本（当前 `0x01`） |
| 1 | 1B | flags | uint8 | 标志位掩码 |
| 2 | 4B | frame_id | uint32 LE | 单调递增帧 ID |
| 6 | 8B | timestamp_ms | int64 LE | Unix 毫秒时间戳 |
| 14 | 4B | scroll_x | int32 LE | 页面水平滚动偏移（CSS px） |
| 18 | 4B | scroll_y | int32 LE | 页面垂直滚动偏移（CSS px） |
| 22 | 2B | viewport_w | uint16 LE | CSS 视口宽度 |
| 24 | 2B | viewport_h | uint16 LE | CSS 视口高度 |
| 26 | 2B | canvas_w | uint16 LE | 物理画布宽度（= viewportW × dpr） |
| 28 | 2B | canvas_h | uint16 LE | 物理画布高度（= viewportH × dpr） |

#### 帧头标志位（flags）

| Bit | 常量 | 含义 |
|-----|------|------|
| 0 | `FLAG_IS_KEYFRAME (0x01)` | 关键帧（需清空画布后重放全部命令） |
| 1 | `FLAG_HAS_FONT_DATA (0x02)` | 帧内含字体内联数据（FONT_DATA 命令） |
| 2 | `FLAG_HAS_DIRTY_RECTS (0x04)` | 帧内含脏区域矩形列表（R-tree 增量） |

#### 序列化注意

由于 `int64_t timestamp_ms` 字段可能导致编译器在结构体中插入 padding，**禁止直接 `memcpy` 整个 FrameHeader**。必须通过 `SerializeFrameHeader()` / `DeserializeFrameHeader()` 逐字段显式序列化。

### 4.3 命令格式

```
┌──────┬──────────┬─────────────────┬──────────┐
│ 1B   │   3B     │    N bytes      │  0-3B    │
│Opcode│ PayLen   │    Payload      │  Padding │
│      │ uint24 LE│                 │ (4B对齐) │
└──────┴──────────┴─────────────────┴──────────┘
```

- **Opcode**: 1 字节，范围 `0x01-0x7F` 合法，`0x80-0xFF` 客户端必须拒收
- **PayLen**: 3 字节小端序 uint24，最大 1MB（`MAX_PAYLOAD_BYTES`）
- **Payload**: 变长，4 字节边界对齐（不足位填充 `0x00`，校验时必须为 0）

| 范围 | 类别 | 说明 |
|------|------|------|
| `0x00` | 保留 | 空操作标记 |
| `0x01-0x0F` | 状态管理 | save, restore, saveLayer |
| `0x10-0x1F` | 变换 | translate, scale, rotate, concat, concat44 |
| `0x20-0x2F` | 裁剪 | clipRect, clipRRect, clipPath |
| `0x30-0x3F` | 形状绘制 | rect, rrect, oval, arc, path, points, region |
| `0x40-0x4F` | 图像绘制 | image, imageRect, lattice, atlas, patch, edgeAA |
| `0x50-0x5F` | 文本绘制 | textBlob, glyphRunList |
| `0x60-0x6F` | 其他绘制 | paint, color, shadow, vertices, drawable, annotation |
| `0x70-0x7F` | 扩展 | fontData, imageData, setMatrix, drawScrollbar, noop |

### 4.6 Payload 编码细节

- 所有多字节值采用 **Little Endian** 字节序
- `f32` 使用 IEEE 754 单精度浮点
- 命令流以 **4 字节对齐**（不足位填充 `0x00`）
- Path verbs 编码: `0=moveTo, 1=lineTo, 2=quadTo, 3=conicTo, 4=cubicTo, 5=close`
- 图像 flag: `0x00=内联传输, 0x01=hash 引用`

| 消息类型 | 方向 | 用途 |
|----------|------|------|

### 5.2 WisonRBIServer — 入口点

> 验证来源: `server/server.js`

```javascript
class WisonRBIServer {
    constructor(options = {})
    // 关键参数:
    //   options.port         — HTTP/WS 端口 (默认 3000)
    //   options.chromiumPath — Chromium 可执行文件路径
    //   options.maxSessions  — 最大并发会话 (默认 4)
    //   options.tlsCert      — TLS 证书路径 (PEM)
    //   options.tlsKey       — TLS 私钥路径 (PEM)
    //   options.tlsCa        — mTLS CA 路径 (可选)

    async start()    // 启动 HTTP + WebSocket 服务
    async shutdown() // 优雅关闭 (SIGTERM/SIGINT 触发, 幂等)
}
```

**消息路由处理流程:**

```
客户端 WebSocket 消息 (JSON)
  → _handleMessage()
    ├─ type=ready            → _handleReady()      → InputProxy.navigate()
    ├─ type=viewport         → InputProxy.updateViewport()
    ├─ type=io               → InputProxy.handleIOEvent()
    └─ type=request_keyframe → 设置 has_pending_keyframe_request
```

**帧路由:**

```
Chromium CDP → sendFrame(meta, commandStream)
  → font_validator()     → 字体 Magic 校验
  → assembleFrame()      → 30B Header + CommandStream + 4B CRC32
  → compressFrame()      → gzip 压缩 + 三层 zip bomb 防护
  → ws.send(binaryFrame) → WebSocket 发送 (背压检测)
```

**HTTP 端点:**

### 6.3 帧接收流水线

> 验证来源: `client/index.js` — `handleFrame()`

```
handleFrame(compressedFrame)
  1. decompressWithProtection()  → 三层 zip bomb 防护解压
  2. validateFrameCRC()          → CRC32 完整性校验
  3. parseFrameHeader()          → 解析 30B 帧头
  4. 版本检查                    → version != PROTOCOL.VERSION → 丢弃 + version_error
  5. extractCommandStream()      → 提取命令流
  6. validator.validate()        → 9 层白名单校验
  7. 渲染调度                    → requestAnimationFrame (最新帧策略)
  8. renderFrame()               → CanvasKit 逐命令重放
```

**最新帧策略:**

- 若上一帧正在渲染（`renderScheduled = true`），新帧覆盖 `pendingFrameId` / `pendingFrameData`
- 渲染循环在 `requestAnimationFrame` 中执行，避免阻塞主线程
- 始终只渲染最新帧，丢弃过时帧（降低延迟，牺牲中间帧）

### 6.4 CommandValidator — 9 层纵深防御

> 验证来源: `client/command_validator.js`

```javascript
class CommandValidator {
    validate(commandsBuffer) → {
        valid: boolean,
        commandCount?: number,
        rejectOffset?: number,
        rejectReason?: string,
        shouldRequestKeyframe?: boolean
    }

    getOpcodeName(opcode) → string  // 0x35 → "DRAW_PATH"
}
```

**9 层校验（由外到内）:**

| 层 | 校验内容 | 威胁模型 |
|----|----------|----------|
| L1 | Opcode 白名单（0x01-0x7F 精确枚举） | 攻击者发送非法 opcode（≥0x80） |
| L2 | Payload 大小 ≤ 1MB（`MAX_PAYLOAD_BYTES`） | 单命令超大 payload OOM |
| L3 | 帧级总字节 ≤ 64MB（`MAX_BYTES_PER_FRAME`） | 恶意帧耗尽内存（v1.6 P0 S1 引入） |
| L4 | Save/Restore 深度配对 | 栈不平衡导致溢出/欠载 |
| L5 | Payload 子结构深度校验 | count 字段伪造 OOM（v1.6 增强） |
| L6 | 缓冲区边界检查 | 越界读取（每条命令不超出帧边界） |
| L7 | 对齐填充全零校验 | 填充字节含隐藏数据 |
| L8 | 命令数量 ≤ 100,000（`MAX_COMMANDS_PER_FRAME`） | 海量微小命令 CPU 耗尽 |
| L9 | Paint Shader 子结构校验 | 内嵌数据绕过检查 |

**连续拒绝策略:**

- 连续拒绝 ≥ 3 帧（`CONSECUTIVE_REJECT_THRESHOLD`） → 发送 `request_keyframe`
- 拒绝计数器在每次校验通过时归零

**子结构上限（L5 深度校验内容）:**

| 子结构 | 上限 | 防护 |
|--------|------|------|
| Path verbs | 100,000（`kMaxPathVerbs`） | 防止恶意 path 分配巨型数组 |
| TextBlob glyphs | 50,000（`kMaxTextBlobGlyphs`） | 防止伪造 glyphCount OOM |
| Vertices count | 100,000（`kMaxVerticesCount`） | 防止伪造 vertexCount OOM |
| Atlas count | 100,000（`kMaxAtlasCount`） | 防止伪造 count OOM |
| Points count | 100,000（`kMaxPointsCount`） | 防止伪造点集 OOM |

### 6.5 FontRegistry — 字体注册表

> 验证来源: `client/font_registry.js`

```javascript
class FontRegistry {
    constructor(canvasKit, options)
    // options.maxBytes — 最大缓存 (默认 64MB = FONT_CACHE_BYTES)

    registerFont(fontId, fontData) → boolean  // 注册字体（前置 Magic 校验）
    getTypeface(fontId) → SkTypeface | null    // 获取字体（无则返回 null）
    invalidate(fontId)                          // 失效字体
    get stats()                                 // { fontCount, currentBytes, maxBytes }

```

**关键不变量:**

- `font_id=0` 固定为 CanvasKit 内置默认字体，**永不驱逐**
- 注册前必须通过 SFNT/WOFF2 Magic 字节白名单校验（5 种格式）
- Magic 校验失败 → 字体拒绝注册，回退到 `font_id=0`
- 字体大小上限 5MB（`MAX_FONT_INLINE_BYTES`），防止内存炸弹
- LRU 驱逐基于单调递增 `_accessCounter`（**非时间戳**，避免 clock 回退问题）

**LRU 驱逐策略:**

- 驱逐时遍历所有条目找 `lastUsed` 最小值（`font_id=0` 跳过）
- O(n) 但 n 最大 ~13（64MB / 5MB），性能无影响
- `font_id=0` 永不驱逐，保证回退字体始终可用

**安全理由:**

`CanvasKit::MakeFreeTypeFaceFromData` 可能触发 FreeType 解析器漏洞；前置 Magic 校验缓解了格式混淆攻击面。

### 6.6 ImageCache — 图像 LRU 缓存

> 验证来源: `client/image_cache.js`

```javascript
class ImageCache {
    constructor(options)
    // options.maxBytes — 最大缓存 (默认 64MB = IMAGE_CACHE_BYTES)

    has(hexHash) → boolean              // 是否存在
    get(hexHash) → ArrayBuffer | null   // 获取（同时 touch）
    put(hexHash, imageData)              // 存入（满则 LRU 驱逐）
    clear()                              // 清空（服务端重启时调用）
    get stats()                          // { entryCount, currentBytes, maxBytes, usage }
}
```

**hash-ref 模式:**

当服务端配置 `--garnet-image-mode=hash-ref` 时：

- 首次出现的图像：内联传输数据 + 32 字节 SHA-256 哈希
- 后续帧：仅发送 32 字节哈希作为引用键
- 客户端通过此模块缓存已接收图像，绘制时直接用哈希查找复用

**安全不变量:**

- 最大容量 64MB（`IMAGE_CACHE_BYTES`），防止攻击者通过大量唯一哈希耗尽客户端内存
- 单个图像 > 64MB 直接拒绝缓存，不影响其他条目
- LRU 驱逐策略保证攻击者无法通过发送新哈希驱逐所有有用数据
- SHA-256 哈希，碰撞计算不可行（2^128 安全级别，经典生日攻击）

**未实现项（Phase 2+）:**

- `crypto.timingSafeEqual` 常量时间哈希比较（`client/image_cache.js` L11 注释，Phase 2+ 实现）

- `'wasm-unsafe-eval'` — CanvasKit WASM 所需
- `object-src 'self'` — 禁止嵌入插件
- `connect-src ws:// wss://` — 同时允许普通与加密 WebSocket，协议由部署方按需选择

#### 请求拦截（双通道）

1. **declarativeNetRequest (DNR) 静态规则** — 主拦截通道
   - `rules.json`: 匹配所有 `http://`/`https://` 导航，重定向到 `index.html?url=XXX`
   - Priority: 1（低于例外站点的 Priority: 100）

2. **tabs.onUpdated 动态兜底** — 备用拦截通道
   - 处理 DNR 无法覆盖的边缘情况
   - 放行规则: 跳过 `chrome-extension://` URL 和 `wison-rbi-bypass` 标记

#### 例外站点管理

- 用户可通过 `chrome.storage.local` 配置例外域名白名单
- 每个例外域名生成一条 DNR `allow` 规则（ID 1000-1999, priority 100）
- 响应 `chrome.storage.onChanged` 实时更新

#### WebSocket 实现

客户端使用**原生浏览器 WebSocket API**（无外部依赖，曾使用 socket.io-client 已移除）。服务端使用 `ws ^8.x`。两端协议兼容。

#### 浏览器快捷键拦截

以下快捷键被拦截，不发往服务端：

- `Ctrl+T` / `Ctrl+W` / `Ctrl+N` / `Ctrl+Q` / `Ctrl+R` 等
- 防止用户意外关闭会话或刷新覆盖服务端状态

## 7. 安全模型

### 7.1 三条核心不变量

1. **不变量 1** — 客户端收到的每个字节只能是合法 Skia 绘制命令（经白名单校验）或帧元数据（frame_id, scroll, viewport）
2. **不变量 2** — 客户端发出的每个字节只能是原始 HID 事件或帧引用（frame_id，不含页面语义）
3. **不变量 3** — 所有页面内容仅在服务端 Chromium 沙箱内执行，不向客户端传输 HTML/CSS/JS

### 7.2 威胁模型与对策

| 威胁 | 对策 |
|------|------|
| 恶意网页 | Chromium 沙箱隔离；客户端不接触原始内容（不变量 3） |
| 信道中间人 | TLS 1.3（AEAD 套件）+ CRC32 完整性校验 |
| 服务端被入侵 | 客户端 CommandValidator 9 层深度校验（不变量 1，纵深防御最后防线） |
| 侧信道攻击 — 像素窃取 | `RecordingCanvas::onReadPixels()` 始终返回 `false`（recording_canvas.h L312） |
| 侧信道攻击 — 时序 | ±1ms 渲染抖动（`CONFIG.JITTER_ENABLED: true`，已启用） |
| gzip zip bomb | 三层纵深防护：压缩大小 ≤4MB / 解压 ≤64MB / 压缩比 ≤1000:1 |
| DoS — 超大帧 | 帧级 64MB 硬上限（`MAX_BYTES_PER_FRAME`） |
| DoS — 海量命令 | 单帧最多 100,000 条命令（`MAX_COMMANDS_PER_FRAME`） |
| 字体二进制外泄 | SFNT/WOFF2 Magic 白名单（5 种格式） |
| 图像缓存投毒 | SHA-256 hash 引用（2^128 碰撞不可行） |
| 进程崩溃循环 | 指数退避重启（1s → 2s → 4s → … → 30s max） |
| 非字体数据伪装 | 字体 Magic 校验失败的字体回退 `font_id=0` |
| 帧号欺骗 | frame_id 单调递增 + 跳跃检测（`FRAME_ID_JUMP_THRESHOLD = 1000`） |
| 重连状态错乱 | 连续 3 帧拒绝 → `request_keyframe` 重新同步 |

### 7.3 安全审计关键点

#### 服务端安全审计

- [ ] Chromium 沙箱是否保持启用（非 `GARNET_UNSAFE_NO_SANDBOX`）
- [ ] TLS 1.3 是否启用（生产环境）
- [ ] CDP 端口是否仅绑定 localhost
- [ ] 压缩炸弹三层防护是否完整（4MB / 64MB / 1000:1）
- [ ] 字体 Magic 白名单是否仅包含 SFNT/WOFF2（5 种）
- [ ] WebSocket payload 上限是否为 64MB
- [ ] AEAD 密码套件是否为唯一允许的套件
- [ ] mTLS 是否在需要的场景启用

#### 客户端安全审计

- [ ] CommandValidator 9 层校验是否全部启用
- [ ] Opcode 白名单是否覆盖所有 0x01-0x7F 值（精确枚举）
- [ ] 子结构深度校验是否检查所有 count 字段（Path/TextBlob/Vertices/Atlas/Points）
- [ ] `readPixels` 是否确实禁用（`onReadPixels` 返回 false）
- [ ] CSP 是否同时允许 ws:// 和 wss:// 连接
- [ ] 浏览器快捷键（Ctrl+T/W/N/Q/R）是否被拦截
- [ ] `JITTER_ENABLED` 是否为 true
- [ ] 字体 Magic 校验失败是否回退 `font_id=0`

### 7.4 纵深防御层次

```
第 1 层: Chromium 进程沙箱（OS 级隔离）
第 2 层: TLS 1.3 信道加密（防窃听/篡改，AEAD 套件）

第 7 层: CSP 资源限制（防第三方代码注入）
第 8 层: 字体 Magic 白名单（防字体二进制外泄通道）
```

## 8. 性能模型

### 8.1 性能指标

> 验证来源: `cpp/garnet_config.h`

| 指标 | 目标 | 实现策略 |
|------|------|---------|
| 帧生成延迟（Compositor 线程） | < 2ms（`kTargetFrameGenLatencyMs`） | Compositor 线程录制 + Worker 线程异步编码 |
| 首帧带宽（gzip 后） | < 1MB（`kTargetFirstFrameBytes`） | 原生 zlib + hash-ref 去重 |
| 增量帧带宽（gzip 后） | < 50KB（`kTargetDeltaFrameBytes`） | R-tree 空间索引裁剪 + hash-ref |
| 客户端 CPU | < 15%（M1 同级, 60fps, `kTargetClientCpuPercent`） | `requestAnimationFrame` 调度 |
| 客户端内存 | < 128MB（`kTargetClientMemoryBytes`） | 字体 LRU (64MB) + 图像 LRU (64MB) |

### 8.2 CRC32 性能

- 预计算 256 条目查找表（1KB，初始化一次）
- O(n) 时间复杂度，逐字节查表
- 多项式：`0xEDB88320`（IEEE 802.3，与 zlib 兼容）
- 64MB 帧 CRC32 计算约 15ms（Node.js），WASM 实现更快（~5ms for 64MB）

### 8.3 gzip 压缩比

| 场景 | 解压大小 | gzip 大小 | 压缩比 |
|------|----------|-----------|--------|
| 纯文本页面 | 500KB | ~80KB | ~6:1 |
| 混合图文页面 | 3MB | ~400KB | ~7.5:1 |
| 纯色大图 | 8MB | ~8MB | ~1:1 |
| 空白页面 | <1KB | ~20B | — |

**zip bomb 阈值:**

- 压缩后 ≤ 4MB（`MAX_COMPRESSED_FRAME`）
- 解压后 ≤ 64MB（`MAX_BYTES_PER_FRAME`）
- 压缩比 ≤ 1000:1（`MAX_COMPRESSION_RATIO`）
- 任一层触发拒绝，丢弃整帧

### 8.4 图像去重效果

**hash-ref 模式节省增量帧带宽: 60-80%**

- 典型场景：页面滚动时，已渲染图像仅发送 32B 哈希
- SHA-256 计算开销：~1ms/MB（服务端，使用 BoringSSL/OpenSSL）
- 客户端命中：`IMAGE_HASH_HITS` 计数器
- 客户端未命中：`IMAGE_HASH_MISSES` 计数器（首次出现需内联传输）

### 9.4 关键实现细节

#### FrameHeader 手动序列化

由于 `int64_t timestamp_ms` 字段可能导致编译器在结构体中插入 padding（保证 8 字节对齐），**禁止直接 `memcpy` 整个 FrameHeader**。必须通过 `SerializeFrameHeader()` / `DeserializeFrameHeader()` 逐字段显式序列化为 30 字节 LE 缓冲区。

#### CommandBuffer 三段式 API

```cpp
buffer_.beginCommand(Opcode::DRAW_RECT);   // 写入 1B opcode + 3B payLen 占位
buffer_.writeRect(rect);                    // 写入 payload 字段
buffer_.writePaint(paint);
buffer_.endCommand();                       // 回填 payLen + 4B 对齐填充
```

- `beginCommand` 写入 4 字节命令头（opcode + uint24 LE payLen 占位）
- payload 写入期间累计字节数
- `endCommand` 回填实际 payLen + 计算 4 字节对齐填充（必须为 `0x00`，客户端 L7 校验）

#### CRC32 实现

- 预计算 256 条目查找表（多项式 `0xEDB88320`）
- IEEE 802.3 反射多项式算法
- 表驱动 O(n) 实现

#### RecordingCanvas — SkCanvas 子类

- 包装 SkCanvas 的 34 个公共方法 + 1 个 `onReadPixels` 覆盖
- `onReadPixels` 始终返回 `false`（安全不变量 I-1，禁止像素回读）
- 非 PictureLayer 图层采集方法（v1.6 P0 A1）：
  - `recordSolidColorLayer` — 纯色背景 → `drawColor` 命令
  - `recordTextureLayer` — WebGL/Canvas → GPU 纹理快照
  - `recordVideoLayer` — `<video>` → 媒体通道（Phase 2）
  - `recordSurfaceLayer` — iframe/跨进程合成表面
  - `recordScrollbarLayer` — 滚动条

- `cc/layers/picture_layer_impl.cc` — UpdateTiles 钩子（+10 行）
- `cc/trees/layer_tree_host_impl.cc` — DrawLayers 钩子（+15 行）
- `cc/BUILD.gn` — 添加 garnet 依赖（+1 行）

**关键风险:**

- R6（HIGH）：字体一致性 — 服务端使用系统字体（如 Noto Sans CJK），客户端 CanvasKit 仅有内置回退字体。同一 glyph ID 在不同字体文件中指向不同字形轮廓。缓解：CanvasKit 内嵌 Noto Sans，服务端完成全部 shaping。
- R5（HIGH）：Chromium M144+ API Breaking Change。缓解：锁定 M143，季度性评估升级窗口。

### 13.2 协议变更流程

当需要添加新 opcode 时，**必须同步更新以下 6 处位置**:

1. `cpp/frame_constants.h` — 添加 `Opcode` 枚举值
2. `client/protocol.js` — 添加 `OPCODE` 条目 + 常量子结构上限
3. `protocol/opcodes.md` — 更新 opcode 文档表
4. `server/config.js` — 如需新的上限常量，添加并 freeze
5. `client/command_validator.js` — 添加子结构深度校验逻辑（L5）
6. `cpp/recording_canvas.h` — 添加对应的 `onDraw*` 虚函数覆盖

**原则**: 所有数值常量必须是四端一致的「单一事实来源」。任一端单独修改常量将导致跨语言边界审计失败。

### 13.3 安全审计检查单
