## 2. 服务端部署

### 2.1 安装

```bash
# 克隆仓库
git clone https://github.com/laolv2023/wison-rbi-B.git
cd wison-rbi-B/server

# 安装依赖（生产模式）
npm ci --production

# 安装 Chromium（Ubuntu/Debian）
sudo apt-get install -y chromium-browser
```

### 2.2 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `PORT` | `3000` | HTTP/WebSocket 监听端口 |
| `CHROMIUM_PATH` | `chromium` | Chromium 可执行文件路径 |
| `MAX_SESSIONS` | `4` | 最大并发会话数 |
| `WISON_TLS_ENABLED` | `0` | 是否启用 TLS（0/1） |
| `WISON_TLS_CERT` | — | TLS 证书路径 |
| `WISON_TLS_KEY` | — | TLS 私钥路径 |
| `WISON_TLS_CA` | — | mTLS 客户端 CA 证书路径 |
| `LOG_LEVEL` | `INFO` | 日志级别（DEBUG/INFO/WARN/ERROR） |
| `GARNET_UNSAFE_NO_SANDBOX` | — | 关闭 Chromium 沙箱（仅开发环境） |

### 2.3 启动

```bash
cd server
export PORT=3000
export CHROMIUM_PATH=/usr/bin/chromium
export WISON_TLS_ENABLED=1
export WISON_TLS_CERT=/path/to/fullchain.pem
export WISON_TLS_KEY=/path/to/privkey.pem
node server.js
```

### 2.4 systemd 服务单元

生产环境推荐使用 systemd 托管服务进程。创建 `/etc/systemd/system/wison-rbi.service`：

```ini
[Unit]
Description=Wison-RBI Browser Isolation Server
After=network.target

[Service]
Type=simple
User=wison-rbi
WorkingDirectory=/opt/wison-rbi/server
ExecStart=/usr/bin/node server.js
Restart=always
RestartSec=5
LimitNOFILE=65536
LimitNPROC=4096
NoNewPrivileges=yes
ProtectSystem=strict
ProtectHome=yes
ReadWritePaths=/tmp
PrivateTmp=yes

[Install]
WantedBy=multi-user.target
```

启用并启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now wison-rbi
sudo systemctl status wison-rbi
sudo journalctl -u wison-rbi -f
```

> **说明**：`NoNewPrivileges`、`ProtectSystem=strict`、`ProtectHome`、`PrivateTmp` 等指令为 systemd 提供的进程级安全加固，可限制服务被入侵后的横向移动能力。如需写入非 `/tmp` 目录（如日志卷），需通过 `ReadWritePaths` 显式声明。

## 3. 客户端部署

### 3.1 加载 Chrome 扩展（MV3）

1. 打开 Chrome，访问 `chrome://extensions`
2. 右上角启用「开发者模式」
3. 点击「加载已解压的扩展程序」
4. 选择仓库中的 `client/` 目录

### 3.2 配置 SERVER_URL

修改 `client/index.js`，将 `CONFIG.SERVER_URL` 指向服务端地址：

```javascript
const CONFIG = Object.freeze({
    SERVER_URL: 'ws://your-server.example.com:3000',  // 默认 ws://，启用 TLS 后改为 wss://
    // ...
});
```

### 3.3 WebSocket 连接说明

> 客户端使用浏览器原生 WebSocket API（无外部依赖），服务端使用 `ws ^8.x`。两端协议兼容，无需额外适配。
>
> 客户端通过 `new WebSocket(url)` 连接服务端，自行实现指数退避重连（1s → 30s）。

### 3.4 企业部署（可选）

可通过 Chrome 策略 `ExtensionInstallForcelist` 强制安装到企业终端：

```json
{
  "ExtensionInstallForcelist": [
    "<extension-id>;https://clients2.google.com/service/update2/crx"
  ]
}
```

## 4. TLS 配置

### 4.1 Let's Encrypt 证书（推荐）

```bash
# 申请证书（standalone 模式，需先停止占用 80 端口的服务）
sudo certbot certonly --standalone -d wison-rbi.example.com

# 配置环境变量
export WISON_TLS_CERT=/etc/letsencrypt/live/wison-rbi.example.com/fullchain.pem
export WISON_TLS_KEY=/etc/letsencrypt/live/wison-rbi.example.com/privkey.pem
```

### 4.2 自签名证书（仅测试用）

```bash
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem \
  -days 365 -nodes -subj "/CN=localhost"
```

### 4.3 mTLS 双向认证（可选）

设置 `WISON_TLS_CA` 启用客户端证书校验：

```bash
export WISON_TLS_CA=/etc/ssl/certs/client-ca.pem
```

### 4.4 TLS 协议参数

| 配置项 | 值 | 说明 |
|--------|------|------|
| 最低版本 | TLSv1.3 | 强制 TLS 1.3，禁用所有旧版本 |
| 密码套件 | TLS_AES_256_GCM_SHA384、TLS_CHACHA20_POLY1305_SHA256、TLS_AES_128_GCM_SHA256 | 仅允许 AEAD 套件 |
| HonorCipherOrder | true | 服务端决定密码套件优先级 |
| mTLS | 可选 | 通过 `WISON_TLS_CA` 配置双向认证 |

## 5. Chromium 沙箱加固

### 5.1 推荐启动参数

```bash
chromium \
  --headless=new \               # 使用新版无头模式
  --disable-gpu \                # 无头模式通常不需要 GPU
  --disable-dev-shm-usage \      # 避免 Docker /dev/shm 空间不足
  --no-first-run \
  --remote-debugging-port=0      # CDP 端口由管理器动态分配
```

### 5.2 沙箱策略

| 参数 | 用途 | 风险等级 |
|------|------|---------|
| `--headless=new` | 新版无头模式（M112+） | 安全 |
| `--disable-gpu` | 无头模式不需要 GPU | 安全 |
| `--disable-dev-shm-usage` | 避免 Docker `/dev/shm` 写满 | 安全 |
| `--no-sandbox` | 关闭 Chromium 沙箱 | **仅开发环境** |

- `--no-sandbox` **仅在开发环境**通过 `GARNET_UNSAFE_NO_SANDBOX=1` 启用
- **生产环境必须保持沙箱开启**
- 沙箱在 Chromium 进程级别隔离页面代码与宿主操作系统，是浏览器隔离体系的第一道防线

### 5.3 崩溃恢复

`chromium_manager.js` 实现了基于指数退避的崩溃恢复机制：

```
1s → 2s → 4s → 8s → 16s → 30s（最大值）
```

- 每次重启间隔按指数增长，避免在持续故障时频繁拉起进程
- 退避上限 30s，避免长时间无服务
- 重启次数无上限，但每次重启会生成新的会话上下文，原会话需要客户端重新连接

## 6. Garnet 模块编译

> **目标**：将 Wison-RBI 的 C++ Garnet 模块集成到 Chromium M143 源码树中编译。
> **Chromium 版本**：143.0.7499.192-1（Stable: 2025-12-02）

### 6.1 环境要求

| 维度 | 最低 | 推荐 |
|------|------|------|
| 操作系统 | Ubuntu 22.04 / Debian 12 | Ubuntu 22.04 LTS |
| CPU | 8 核 | 16+ 核 |
| 内存 | 16 GB | 32 GB |
| 磁盘 | 120 GB | 200 GB SSD |
| 网络 | 可访问 `chromium.googlesource.com` | 翻墙/代理 |
| 编译时间 | ~4 小时（16 核） | ~2 小时（32 核） |

**资源占用说明**：
- **磁盘 120 GB**：Chromium 源码 ~40 GB + 第三方依赖 ~20 GB + 编译产物 ~60 GB
- **内存 16 GB**：Skia 编译峰值 ~8 GB + Chromium 链接器 ~4 GB + 系统 ~4 GB
- **CPU 8 核**：`ninja -j8` 并行编译，低于 4 核会非常慢

### 6.2 三种编译方式

#### 方式 A — 一键脚本（推荐）

```bash
# 1. 克隆仓库
git clone https://github.com/laolv2023/wison-rbi-B.git
cd wison-rbi-B

# 2. 赋予执行权限
chmod +x scripts/build_chromium.sh

# 3. 一键编译（自动下载源码 + 编译 + 运行测试）
bash scripts/build_chromium.sh --run-tests

# 4. 查看产物
ls -la /opt/chromium/src/out/Garnet/obj/garnet/
```

#### 方式 B — Docker（环境隔离）

```bash
# 1. 克隆仓库
git clone https://github.com/laolv2023/wison-rbi-B.git
cd wison-rbi-B

# 2. 构建镜像
docker compose -f docker/docker-compose.yml build

# 3. 运行编译
docker compose -f docker/docker-compose.yml run --rm chromium-builder --run-tests

# 4. 查看产物
ls -la docker/chromium-build/src/out/Garnet/obj/garnet/
```

#### 方式 C — 手动步骤

```bash
# 1. 安装 depot_tools
git clone https://chromium.googlesource.com/chromium/tools/depot_tools.git /opt/depot_tools
export PATH="/opt/depot_tools:$PATH"

# 2. 下载 Chromium M143 源码
mkdir -p /opt/chromium && cd /opt/chromium
fetch --nohooks chromium
cd src
git checkout tags/143.0.7499.192
gclient sync --no-history -D

# 3. 放置 garnet 模块
cp /path/to/wison-rbi-B/cpp/*.h      /opt/chromium/src/garnet/
cp /path/to/wison-rbi-B/cpp/*.cpp    /opt/chromium/src/garnet/
cp /path/to/wison-rbi-B/cpp/BUILD.gn /opt/chromium/src/garnet/

# 4. 生成构建文件
gn gen out/Garnet --args='is_debug=false is_component_build=false'

# 5. 编译
ninja -C out/Garnet garnet
```

### 6.3 脚本参数

`scripts/build_chromium.sh` 支持以下参数：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `--skip-fetch` | 跳过源码下载（已下载时使用） | 否 |
| `--skip-build` | 跳过编译（仅放置 garnet 模块） | 否 |
| `--run-tests` | 编译后运行测试 | 否 |
| `--jobs N` | 编译并行数 | `$(nproc)` |
| `--chromium-src PATH` | Chromium 源码路径 | `/opt/chromium/src` |
| `--help` | 显示帮助 | — |

**常见用法**：

```bash
# 首次编译（下载源码 + 编译 + 测试）
bash scripts/build_chromium.sh --run-tests

# 已有源码，仅重新放置 garnet 并编译
bash scripts/build_chromium.sh --skip-fetch

# 仅放置 garnet 模块，不编译
bash scripts/build_chromium.sh --skip-fetch --skip-build

# 指定源码路径和并行数
bash scripts/build_chromium.sh --chromium-src /data/chromium/src --jobs 32

# Docker 中运行
docker compose -f docker/docker-compose.yml run --rm chromium-builder --run-tests
docker compose -f docker/docker-compose.yml run --rm chromium-builder --skip-fetch --jobs 16
```

### 6.4 条件编译机制

Garnet 模块通过宏定义切换 Mock Skia 与真实 Skia：

```
┌─────────────────────────────────────────────────────────────┐
│  garnet_standalone.h                                        │
│                                                             │
│  #ifdef GARNET_STANDALONE  ← 独立编译/测试                  │
│    #include "test_mocks.h"    ← Mock Skia 类型              │
│  #else                      ← Chromium 集成                 │
│    #include "include/core/SkCanvas.h"  ← 真实 Skia          │
│    #include "include/core/SkPath.h"                         │
│    ...                                                      │
│  #endif                                                     │
└─────────────────────────────────────────────────────────────┘
```

| 编译模式 | 宏定义 | Skia 来源 | 用途 |
|---------|--------|----------|------|
| **独立编译** | `GARNET_STANDALONE` | `test_mocks.h`（Mock） | 单元测试、CI |
| **Chromium 集成** | `GARNET_INTEGRATED` | `//skia`（真实） | 生产环境 |

#### 独立编译（测试用）

```bash
g++ -std=c++20 -DGARNET_STANDALONE -Icpp/ \
    cpp/command_buffer.cpp \
    cpp/layer_recorder.cpp \
    cpp/recording_canvas.cpp \
    cpp/test_runner.cpp \
    -o garnet_tests && ./garnet_tests
```

**预期结果**：120 个测试函数，1546 项断言，**1543 PASS / 3 FAIL**。
3 个失败用例与 Mock `encodeToData` 行为相关，不影响 Chromium 集成编译。

### 6.5 BUILD.gn 说明

```gn
# cpp/BUILD.gn
static_library("garnet") {
  sources = [
    "command_buffer.cpp",      # 命令缓冲区序列化
    "layer_recorder.cpp",      # 图层录制器
    "recording_canvas.cpp",    # SkCanvas 子类（录制绘制调用）
  ]
  defines = [ "GARNET_INTEGRATED" ]   # 使用真实 Skia
  deps = [ "//skia" ]                 # 依赖 Chromium 内置 Skia
}
```

### 6.6 Compositor 挂载（编译完成后的下一步）

编译通过后，需修改 Chromium Compositor 代码挂载 garnet：

```cpp
// 在 //cc/layer_tree_host.cc 的 DidCommit() 中:
#include "garnet/layer_recorder.h"

void LayerTreeHost::DidCommit() {
    // ... 原有逻辑 ...
    if (garnet_recorder_) {
        garnet_recorder_->RecordFrame(root_layer());
    }
}
```

> **注意**：Compositor 挂载点的修改需在 Chromium 源码树中完成，不在 `build_chromium.sh` 范围内。
> 完整方案见 `AUDIT_AND_IMPL_PLAN_M143.md` 第 7 节「Compositor 挂载方案」。

### 6.7 故障排除

#### 源码下载失败

```
fatal: unable to access 'https://chromium.googlesource.com/...'
```

**解决方案**：

```bash
# 使用代理
export http_proxy=http://127.0.0.1:7890
export https_proxy=http://127.0.0.1:7890
```

#### 编译内存不足（OOM）

```
ninja: fatal: Cannot allocate memory
```

**解决方案**：

```bash
# 减少并行数
bash scripts/build_chromium.sh --jobs 4

# 或增加 swap
sudo fallocate -l 16G /swapfile
sudo mkswap /swapfile && sudo swapon /swapfile
```

#### Skia API 不兼容

```
error: no matching function for call to 'SkCanvas::drawPicture'
```

**原因**：Mock Skia 与真实 Skia M143 的 API 签名可能存在差异。

**解决方案**：
1. 检查 `cpp/test_mocks.h` 中对应函数的签名
2. 对照 `//skia/include/core/SkCanvas.h` 的实际签名
3. 修改 `recording_canvas.cpp` 适配真实 API
4. 参考 `AUDIT_AND_IMPL_PLAN_M143.md` 第 8 节「待验证项 [U-1]~[U-5]」

#### 磁盘空间不足

```bash
# 检查空间
df -h /opt/chromium

# 清理编译产物（保留源码）
rm -rf /opt/chromium/src/out/Garnet
```

#### Docker 权限问题

```bash
sudo usermod -aG docker $USER
# 重新登录后重试
```

## 7. 监控与告警

### 7.1 健康检查端点

```
GET /health   → {"status":"ok","sessions":2,"uptime":3600.5,"tls":true}
GET /metrics  → JSON 完整指标快照（counter / gauge / histogram）
```

**使用示例**：

```bash
# 基础健康检查
curl https://localhost:3000/health

# 完整指标
curl https://localhost:3000/metrics
```

### 7.2 关键指标

| 指标 | 类型 | 说明 |
|------|------|------|
| `sessions_active` | gauge | 当前活跃会话数 |
| `sessions_total` | counter | 累计会话数 |
| `frames_sent` | counter | 累计发送帧数 |
| `frames_dropped` | counter | 累计丢弃帧数 |
| `ws_connections` | gauge | 当前 WebSocket 连接数 |
| `ws_backpressure` | gauge | WebSocket 背压状态 |
| `ws_errors` | counter | WebSocket 错误数 |
| `unhandled_rejections` | counter | 未处理 Promise 拒绝数 |

### 7.3 告警规则

| 规则 | 条件 | 严重级别 |
|------|------|----------|
| 高帧丢弃率 | drop / send > 10% | WARN |
| CRC 失败异常 | fail / send > 5% | CRITICAL |
| 会话饱和 | active ≥ limit | WARN |
| 高频背压 | > 5 次/分钟 | WARN |

### 7.4 日志审计

- **会话创建/销毁**：含 `sessionId`、客户端 IP
- **帧发送**：`frameId`、`compressedSize`、`commandCount`
- **输入事件**：不含页面语义，仅坐标/按键类型
- **安全事件**：CRC 失败、opcode 拒绝、zip bomb 防护触发

```bash

```

## 9. 故障排查

### 9.1 常见问题

| 症状 | 可能原因 | 解决方案 |
|------|---------|---------|
| 客户端白屏 | CanvasKit 加载失败 | 检查 CDN/本地路径，确认 MIME 类型 |
| 连接立即断开 | TLS 证书无效 | 检查证书路径，确认客户端信任 CA |
| 画面不更新 | 帧被丢弃 | 检查 `MAX_COMPRESSED_FRAME`、网络带宽 |
| Chromium 频繁重启 | OOM | 增加服务器内存 / 降低 `MAX_SESSIONS` |
| 增量帧异常 | `frame_id` 不连续 | 检查是否触发 `request_keyframe` |
| 字体显示错乱 | 字体 Magic 校验失败 | 检查 `font_validator.js` 日志 |

## 10. 已知问题与风险

### 10.1 WebSocket 库不兼容 — ✅ 已修复

> **原问题**：客户端曾使用 `socket.io-client ^4.7.4`，服务端使用 `ws ^8.x`。socket.io 在 WebSocket 之上增加自定义协议层（Engine.IO），无法直接连接 `ws` 服务器。

**状态**：✅ **已修复**

**修复内容**：
- `client/index.js`：`io()` → `new WebSocket()`，`socket.emit()` → `socket.send(JSON.stringify())`，`socket.on('frame', ...)` → `socket.onmessage`，自行实现指数退避重连
- `client/package.json`：移除 `socket.io-client` 依赖
- `client/manifest.json`：移除 `socket.io-client` 资源声明
- `server/server.js`：无需修改（已使用 `ws` 库）

### 10.2 C++ 层未集成到 Chromium — ❌ 待完成

> **问题描述**：`cpp/` 目录下的代码是独立的 C++ 实现，包含 Mock 测试（`test_mocks.h`），但未实际集成到 Chromium 源码树中。

**影响**：当前系统仅能通过 CDP（Chrome DevTools Protocol）控制无头 Chromium，**无法实现真正的 Compositor 层拦截**。

**解决方案**：按照第 6 节流程，将 C++ 代码集成到 Chromium M143 源码树中编译。编译基础设施（脚本、Docker、BUILD.gn）已就绪，需在 Chromium 源码树中完成挂载。
