## 1. 系统概述

### 1.1 定位

Wison-RBI 是一个**浏览器隔离（Browser Isolation）**系统。通过在 Chromium Compositor 层拦截绘制命令，使客户端浏览器不必执行任何远程 HTML/CSS/JavaScript，仅接收经过安全校验的 Skia 绘制命令。

**来源**: README.md 第1~10行；docs/Wison-RBI-技术文档.md 第27~43行。

### 1.2 核心原理

```
传统网络浏览:
  客户端浏览器 ← HTTP → Web 服务器
  ↑ 客户端执行所有 HTML/CSS/JS

Wison-RBI:
  客户端 ← WebSocket(二进制帧) ← 服务端 Chromium ← HTTP → Web 服务器
  ↑ 客户端仅执行 Skia 绘制命令（不含 HTML/CSS/JS）
```

**来源**: docs/Wison-RBI-技术文档.md 第35~43行。

### 1.3 与 Cloudflare NVR 的关键区别

| 维度 | Cloudflare NVR | Wison-RBI |
|------|---------------|-----------|
| 拦截层级 | Skia API | Compositor 层 (cc::DisplayItemList/PaintOp) |
| Chromium 修改范围 | ~15 个文件 | 1-2 个文件 + 独立 RecordingCanvas |
| 客户端引擎 | 专有 WASM | CanvasKit WASM (开源标准 Skia) |
| 增量更新 | 未公开 | R-tree 空间索引驱动 |
| 可审计性 | 闭源 | 全链路开源 |

**来源**: README.md 第14~24行。

**来源**: docs/Wison-RBI-技术文档.md 第3.1节；实际代码模块关系分析。

### 3.2 数据流

```
[页面渲染]
Chromium Compositor → RecordingCanvas (拦截绘制调用)
  → CommandBuffer (序列化为二进制命令流)
  → LayerRecorder + FrameAssembler (组装帧)
  → frame_builder.js (30B Header + Commands + 4B CRC32 + gzip)
  → WebSocket (wss://)
  → 客户端 index.js (解压 + CRC校验)
  → command_validator.js (9层安全校验)
  → CanvasKit WASM (Skia 渲染)
  → 用户屏幕

[用户输入]
用户鼠标/键盘 → 客户端 index.js (HID 事件)
  → WebSocket (JSON)
  → server.js → io_proxy.js (坐标转换)
  → CDP Input.dispatchMouseEvent / dispatchKeyEvent
  → Chromium 页面响应
```

**来源**: server/server.js 第18~30行；client/index.js 模块角色注释。

## 4. 安全模型

### 4.1 三条核心不变量

1. **客户端收到的每个字节**只能是：合法 Skia 绘制命令（经白名单校验）或帧元数据（frame_id, scroll, viewport）
2. **客户端发出的每个字节**只能是：原始 HID 事件或帧引用（frame_id，不含页面语义）
3. **所有页面内容**仅在服务端 Chromium 沙箱内执行，不向客户端传输 HTML/CSS/JS

**来源**: README.md 第128~134行；docs/Wison-RBI-技术文档.md 第7节。

### 4.2 威胁模型与对策

| 威胁 | 对策 | 实现位置 |
|------|------|---------|
| 恶意网页 | Chromium 沙箱隔离；客户端不接触原始内容 | `chromium_manager.js` |
| 信道中间人 | TLS 1.3 + 命令格式校验 + CRC32 | `server/config.js` TLS配置; `frame_builder.js` CRC32 |
| 服务端被入侵 | 客户端命令白名单 + 参数范围校验 | `command_validator.js` 9层校验 |
| 侧信道攻击 | readPixels 禁用；±1ms 渲染抖动 (Phase 2) | `recording_canvas.cpp` |
| gzip zip bomb | 三层防护: 压缩大小/流式解压/压缩比异常检测 | `garnet_config.h` + `utils.js` |

**来源**: README.md 第136~142行；cpp/garnet_config.h 第64~93行。

### 4.3 客户端命令校验（9层纵深防御）

| 层级 | 校验内容 | 实现位置 |
|------|---------|---------|
| 1 | Opcode 白名单: 仅允许 0x01-0x7F | `command_validator.js` |
| 2 | Payload 大小: ≤ 1MB | `command_validator.js` |
| 3 | 帧级总字节: ≤ 64MB | `command_validator.js` |
| 4 | Save/Restore 配对: 深度计数平衡 | `command_validator.js` |
| 5 | Payload 子结构深度校验 | `command_validator.js` |
| 6 | 缓冲区边界: 不超出帧边界 | `command_validator.js` |
| 7 | 对齐填充校验: 填充字节全零 | `command_validator.js` |
| 8 | 命令数量上限: ≤ 100,000 | `command_validator.js` |
| 9 | Paint 内 Shader 子结构校验 | `command_validator.js` |

**来源**: client/command_validator.js 第15~24行注释。

### 4.4 字体格式校验

仅接受 SFNT (TrueType/OpenType/Collection) 或 WOFF2 格式。Magic 字节白名单:

| Magic 值 | 说明 |
|----------|------|
| `0x00010000` | TrueType |
| `OTTO` (0x4F54544F) | OpenType CFF |
| `true` (0x74727565) | TrueType (Apple) |
| `ttcf` (0x74746366) | TrueType Collection |
| `wOF2` (0x774F4632) | WOFF2 |

校验失败 → 替换为 CanvasKit 回退字体 (font_id=0)。

**来源**: server/config.js 第137~142行；server/font_validator.js；client/font_registry.js。

## 5. 通信协议

### 5.1 帧格式（二进制）

```
[30B Header] [N B CommandStream] [4B CRC32]

CRC32: Header + CommandStream，多项式 0xEDB88320 (IEEE 802.3)
```

**来源**: README.md 第148~152行；cpp/frame_constants.h。

### 5.2 帧头结构（30字节）

| 偏移 | 大小 | 字段 | 说明 |
|------|------|------|------|
| 0 | 1B | version | 协议版本 (0x01) |
| 1 | 4B | frame_id | 帧ID（单调递增） |
| 5 | 4B | width | 画布宽度 |
| 9 | 4B | height | 画布高度 |
| 13 | 4B | scroll_x | 水平滚动偏移 |
| 17 | 4B | scroll_y | 垂直滚动偏移 |
| 21 | 4B | timestamp | 帧时间戳 (ms) |
| 25 | 4B | command_count | 命令数量 |
| 29 | 1B | flags | 标志位（bit0: keyframe） |

**来源**: cpp/frame_constants.h FrameHeader 结构体；server/config.js FRAME_HEADER_SIZE=30。

### 5.3 命令格式

```
[1B Opcode] [3B PayloadLength (uint24 LE)] [N B Payload]

4 字节对齐（填充零字节）
```

**来源**: protocol/opcodes.md；cpp/frame_constants.h。

### 5.4 Opcode 分配

| 范围 | 类别 | 说明 |
|------|------|------|
| `0x00` | 保留 | 空操作标记 |
| `0x01-0x0F` | 状态管理 | save, restore, saveLayer |
| `0x10-0x1F` | 变换 | translate, scale, rotate, concat, concat44 |
| `0x20-0x2F` | 裁剪 | clipRect, clipRRect, clipPath |
| `0x30-0x3F` | 形状绘制 | rect, rrect, oval, arc, path, points, region |
| `0x40-0x4F` | 图像绘制 | image, imageRect, lattice, atlas, patch, edgeAA |
| `0x50-0x5F` | 文本绘制 | textBlob, glyphRunList |
| `0x60-0x6F` | 其他绘制 | paint, color, shadow, vertices, drawable, annotation |
| `0x70-0x7F` | 扩展 | fontData, imageData, setMatrix, drawScrollbar, noop |
| `0x80-0xFF` | **非法** | 客户端必须拒收 |

**来源**: protocol/opcodes.md；cpp/frame_constants.h；client/protocol.js。

### 5.5 安全硬上限

| 常量 | 值 | 说明 | 来源 |
|------|------|------|------|
| `MAX_BYTES_PER_FRAME` | 64 MB | 单帧解压后总字节上限 | garnet_config.h 第75行 |
| `MAX_COMPRESSED_FRAME` | 4 MB | gzip 压缩后帧大小上限 | garnet_config.h 第84行 |
| `MAX_COMPRESSION_RATIO` | 1000:1 | 压缩比异常阈值 | garnet_config.h 第93行 |
| `MAX_PAYLOAD_BYTES` | 1 MB | 单命令 Payload 上限 | garnet_config.h |
| `MAX_COMMANDS_PER_FRAME` | 100,000 | 单帧命令数量上限 | garnet_config.h |
| `MAX_FONT_INLINE_BYTES` | 5 MB | 单字体内联大小上限 | client/font_registry.js |
| `FONT_CACHE_BYTES` | 64 MB | 字体缓存总容量 | client/font_registry.js |
| `IMAGE_CACHE_BYTES` | 64 MB | 图像缓存总容量 | client/image_cache.js |

## 6. 已知问题与风险

### 6.1 WebSocket 库不兼容（高优先级）

> **问题描述**: 客户端使用 `socket.io-client ^4.7.4`，服务端使用 `ws ^8.x`。socket.io 在 WebSocket 之上增加了自定义协议层（Engine.IO），无法直接连接 `ws` 服务器。

**影响**: 客户端无法连接服务端，系统无法正常运行。

**来源**: 
- client/package.json 依赖 `socket.io-client`
- server/package.json 依赖 `ws`
- client/index.js 第227行注释 "使用 socket.io 库"
- server/server.js 第54行 `require('ws')`

**解决方案（二选一）**:
1. **服务端改用 socket.io**: `npm install socket.io`，将 `server.js` 中的 `WsServer` 替换为 `io(server)`
2. **客户端改用原生 WebSocket**: 移除 `socket.io-client` 依赖，`client/index.js` 中使用 `new WebSocket(url)` 并自行实现重连逻辑

**建议**: 方案2更轻量，减少依赖，推荐采用。

### 6.2 C++ 层未集成到 Chromium（中优先级）

> **问题描述**: `cpp/` 目录下的代码是独立的 C++ 实现，包含 Mock 测试（`test_mocks.h`），但未实际集成到 Chromium 源码树中。

**影响**: 当前系统仅能通过 CDP（Chrome DevTools Protocol）控制无头 Chromium，无法实现真正的 Compositor 层拦截。

**来源**: cpp/garnet_config.h 第10~13行注释 "挂载路径: //garnet/garnet_config.h"；cpp/test_mocks.h 使用 Mock 而非真实 Skia。

**解决方案**: 按照 `AUDIT_AND_IMPL_PLAN_M143.md` 的方案，将 C++ 代码集成到 Chromium M143 源码树中编译。

### 6.3 Phase 2/3 功能未实现（低优先级）

> **问题描述**: 部分高级功能标注为 Phase 2/3，当前未实现:
> - ±1ms 渲染抖动（侧信道防御）
> - crypto.timingSafeEqual 常量时间比较
> - 完整的 SkImageFilter 子树序列化

**来源**: README.md 第140行 "Phase 2"；client/image_cache.js 第18行 "Phase 2+"。

### 7.1 系统要求

#### 服务端

| 资源 | 最低 | 推荐 |
|------|------|------|
| CPU | 4 核 (x86_64) | 8+ 核 |
| 内存 | 8 GB | 16 GB+ (每并发会话 +2GB) |
| 磁盘 | 20 GB | SSD 50 GB+ |
| 操作系统 | Linux (kernel ≥5.4) | Ubuntu 22.04 LTS / Debian 12 |
| Node.js | ≥18.0 | 20.x LTS |
| Chromium | ≥M120 (headless) | 与 CanvasKit 同 Milestone |

#### 客户端

| 资源 | 要求 |
|------|------|
| 浏览器 | Chrome/Chromium ≥ M120 (支持 MV3 + WebCodecs) |
| 内存 | ≥512 MB 可用 (WASM heap) |
| GPU | WebGL 2.0 支持 |
| 网络 | 稳定连接，延迟 <100ms RTT 推荐 |

**来源**: DEPLOYMENT.md 第18~38行。

### 7.2 服务端部署

```bash
# 克隆仓库
git clone https://github.com/laolv2023/wison-rbi-B.git
cd wison-rbi-B/server

# 安装依赖
npm ci --production

# 安装 Chromium
sudo apt-get install -y chromium-browser

# 配置环境变量
export PORT=3000
export CHROMIUM_PATH=/usr/bin/chromium
export MAX_SESSIONS=4

# TLS (生产环境必须)
export WISON_TLS_ENABLED=1
export WISON_TLS_CERT=/etc/ssl/certs/wison-rbi-fullchain.pem
export WISON_TLS_KEY=/etc/ssl/private/wison-rbi-privkey.pem

# 启动
node server.js
```

**来源**: DEPLOYMENT.md 第42~83行。

### 7.3 客户端部署

```bash
cd wison-rbi-B/client
npm ci --production
```

**加载扩展**:
1. 打开 `chrome://extensions`
2. 启用"开发者模式"
3. 点击"加载已解压的扩展程序"
4. 选择 `client/` 目录

**配置扩展**: 修改 `client/index.js` 中的 `CONFIG.SERVER_URL`:
```javascript
const CONFIG = Object.freeze({
    SERVER_URL: 'wss://your-server.example.com:3000',
});
```

**来源**: DEPLOYMENT.md 第127~152行。

### 7.4 TLS 配置

| 配置项 | 值 | 说明 |
|--------|------|------|
| 最低版本 | TLSv1.3 | 强制 TLS 1.3，禁用所有旧版本 |
| 密码套件 | TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256, TLS_AES_128_GCM_SHA256 | 仅允许 AEAD 套件 |
| HonorCipherOrder | true | 服务端决定密码套件优先级 |
| mTLS | 可选 | 通过 WISON_TLS_CA 配置双向认证 |

**来源**: server/metrics.js；server/server.js。

### 8.2 关键指标

| 指标 | 类型 | 说明 |
|------|------|------|
| `sessions_active` | gauge | 当前活跃会话数 |
| `sessions_total` | counter | 累计会话数 |
| `frames_sent` | counter | 累计发送帧数 |
| `frames_dropped` | counter | 累计丢弃帧数 |
| `ws_connections` | gauge | 当前 WebSocket 连接数 |
| `ws_backpressure` | gauge | WebSocket 背压状态 |
| `ws_errors` | counter | WebSocket 错误数 |
| `unhandled_rejections` | counter | 未处理 Promise 拒绝数 |

**来源**: server/metrics.js；server/server.js 第61~72行。

### 8.3 告警阈值

| 告警条件 | 说明 |
|----------|------|
| `frames_dropped / frames_sent > 5%` | 帧丢弃率过高 |
| `ws_backpressure > 0` | WebSocket 背压 |
| `sessions_active > MAX_SESSIONS * 0.8` | 会话数接近上限 |

**来源**: server/metrics.js checkAlerts()。

## 14. 结论

Wison-RBI v1.6 是一个架构清晰的浏览器隔离系统，具有以下特点:

**优势**:
- Compositor 层拦截设计，Chromium 修改范围小（1-2个文件）
- 客户端使用开源 CanvasKit WASM，无需专有引擎
- 9层纵深防御的命令校验体系
- 完整的测试覆盖（200+ Node.js 用例 + 120 C++ 用例）
- 全链路开源，可审计

**风险**:
- WebSocket 库不兼容问题需在投运前解决（高优先级）
- C++ 层尚未集成到 Chromium 源码树（中优先级）
- Phase 2/3 高级功能未实现（低优先级）

**建议**: 解决 WebSocket 库兼容性问题后，可进行小规模试点投运。完整投运需完成 C++ 层集成到 Chromium 源码树。
