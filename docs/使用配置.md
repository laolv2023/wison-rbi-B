## 1. 概述

### 1.1 项目定位

Wison-RBI 是一个**浏览器隔离（Browser Isolation, RBI）**系统。通过在 Chromium Compositor 层拦截绘制命令，使客户端浏览器不必执行任何远程 HTML/CSS/JavaScript，仅接收经过安全校验的 Skia 绘制命令。

**核心原理**:

```
传统网络浏览:
  客户端浏览器 ← HTTP → Web 服务器
  ↑ 客户端执行所有 HTML/CSS/JS（暴露于所有客户端侧漏洞）

Wison-RBI:
  客户端 ← WebSocket(二进制帧) ← 服务端 Chromium ← HTTP → Web 服务器
  ↑ 客户端仅执行 Skia 绘制命令（不含 HTML/CSS/JS）
  ↑ 所有页面内容仅在服务端 Chromium 沙箱内执行
```

### 1.2 三条核心安全不变量

1. **客户端收到的每个字节**只能是：合法 Skia 绘制命令（经白名单校验）或帧元数据（frame_id, scroll, viewport）
2. **客户端发出的每个字节**只能是：原始 HID 事件或帧引用（frame_id，不含页面语义）
3. **所有页面内容**仅在服务端 Chromium 沙箱内执行，不向客户端传输 HTML/CSS/JS

### 1.3 与 Cloudflare NVR 的关键区别

| 维度 | Cloudflare NVR | Wison-RBI |
|------|---------------|-----------|
| 拦截层级 | Skia API | Compositor 层 (cc::DisplayItemList/PaintOp) |
| Chromium 修改范围 | ~15 个文件 | 1-2 个文件 + 独立 RecordingCanvas |
| 客户端引擎 | 专有 WASM | CanvasKit WASM (开源标准 Skia) |
| 增量更新 | 未公开 | R-tree 空间索引驱动 |
| 可审计性 | 闭源 | 全链路开源 |

### 1.5 数据流

```
[页面渲染 → 客户端显示]
Chromium Compositor → RecordingCanvas (拦截绘制调用)
  → CommandBuffer (序列化为二进制命令流)
  → LayerRecorder + FrameAssembler (组装帧)
  → frame_builder.js (30B Header + Commands + 4B CRC32 + gzip)
  → WebSocket (wss://)
  → 客户端 index.js (解压 + CRC校验)
  → command_validator.js (9层安全校验)
  → CanvasKit WASM (Skia 渲染)
  → 用户屏幕

[用户输入 → 服务端页面响应]
用户鼠标/键盘 → 客户端 index.js (HID 事件)
  → WebSocket (JSON 消息: type=io)
  → server.js → io_proxy.js (坐标转换)
  → CDP Input.dispatchMouseEvent / dispatchKeyEvent
  → Chromium 页面响应
```

## 2. 快速开始

### 2.1 前置条件

| 组件 | 服务端要求 | 客户端要求 |
|------|-----------|-----------|
| 操作系统 | Linux (kernel ≥5.4)，推荐 Ubuntu 22.04 LTS / Debian 12 | Windows / macOS / Linux |
| Node.js | ≥ 18.0（推荐 20.x LTS） | 不适用 |
| 浏览器 | 不适用 | Chrome/Chromium ≥ M120（支持 MV3） |
| Chromium | ≥ M120（headless） | 不适用 |
| 内存 | ≥ 8 GB（每并发会话 +2 GB） | ≥ 512 MB 可用 (WASM heap) |
| GPU | 不强制 | WebGL 2.0 支持 |

### 2.2 服务端快速开始

```bash
# 1. 安装依赖
cd server && npm ci --production

# 2. 配置 Chromium 可执行路径
export CHROMIUM_PATH=/usr/bin/chromium

# 3. 启动服务（默认监听 3000 端口）
node server.js
```

启动成功标志:
- 控制台输出 `WebSocket server listening on port 3000`
- 健康检查端点 `http://localhost:3000/health` 返回 200

### 2.3 客户端快速开始

客户端为 Chrome MV3 扩展，无需构建步骤:

1. 打开 `chrome://extensions`
2. 启用右上角的"开发者模式"
3. 点击"加载已解压的扩展程序"
4. 选择项目根目录下的 `client/` 目录

加载完成后，需要修改 `client/index.js` 中的 `CONFIG.SERVER_URL`:

```javascript
const CONFIG = Object.freeze({
    SERVER_URL: 'wss://localhost:3000',  // 修改为实际服务端地址
    // ...
});
```

> **注意**: 客户端使用浏览器原生 WebSocket API（无 socket.io-client 依赖），服务端使用 `ws ^8.x` 库。两端协议兼容，无需额外适配。

### 2.4 首次运行验证

```bash
# 步骤 1: 验证服务端健康
curl http://localhost:3000/health
# 预期输出: {"status":"ok","sessions":0,"uptime":N,"tls":false}

# 步骤 2: 验证指标端点
curl http://localhost:3000/metrics
# 预期输出: JSON，包含 uptime_seconds、counters、gauges、histograms

# 步骤 3: 验证 Chromium 已就绪
ps aux | grep -E 'chromium.*--headless' | grep -v grep
# 预期输出: 至少一个 chromium 进程

# 步骤 4: 客户端连接测试
# 在 Chrome 扩展页面点击 Wison-RBI 图标
# 打开开发者工具 Console，应看到:
#   ws_connected
#   canvaskit_loaded
#   init_complete
```

## 3. 服务端配置

### 3.1 环境变量

服务端通过环境变量进行运行时配置。所有环境变量在 `server/config.js` 中读取。

| 变量名 | 默认值 | 说明 |
|--------|--------|------|
| `PORT` | `3000` | HTTP/WebSocket 服务监听端口 |
| `CHROMIUM_PATH` | `chromium` | Chromium 可执行文件路径 |
| `MAX_SESSIONS` | `4` | 最大并发会话数 |
| `WISON_TLS_ENABLED` | `0` (false) | 是否启用 TLS（1=启用） |
| `WISON_TLS_CERT` | — | TLS 证书路径 |
| `WISON_TLS_KEY` | — | TLS 私钥路径 |
| `WISON_TLS_CA` | — | mTLS 双向认证 CA 证书路径（可选） |
| `LOG_LEVEL` | `INFO` | 日志级别: `DEBUG` / `INFO` / `WARN` / `ERROR` |
| `GARNET_UNSAFE_NO_SANDBOX` | (未设置) | 禁用 Chromium 沙箱（仅开发环境，**生产环境严禁启用**） |

#### 典型配置示例

**开发环境**:
```bash
export PORT=3000
export CHROMIUM_PATH=/usr/bin/chromium
export MAX_SESSIONS=2
export LOG_LEVEL=DEBUG
```

**生产环境**:
```bash
export PORT=3000
export CHROMIUM_PATH=/usr/bin/chromium
export MAX_SESSIONS=4
export LOG_LEVEL=INFO
export WISON_TLS_ENABLED=1
export WISON_TLS_CERT=/etc/ssl/certs/wison-rbi-fullchain.pem
export WISON_TLS_KEY=/etc/ssl/private/wison-rbi-privkey.pem
export WISON_TLS_CA=/etc/ssl/certs/client-ca.pem  # 可选，启用 mTLS
```

> ⚠️ **安全警告**: `GARNET_UNSAFE_NO_SANDBOX` 仅在受信任的开发环境中使用。生产环境启用此变量将完全消除 Chromium 沙箱保护，使宿主机直接暴露于恶意网页攻击面。

### 3.2 服务端配置常量

以下常量在 `server/config.js` 中通过 `Object.freeze()` 冻结，运行时不可篡改。修改需编辑源码并重启服务。

#### 3.2.1 协议常量

```javascript
PROTOCOL.VERSION              = 0x01          // 协议版本号
PROTOCOL.FRAME_HEADER_SIZE    = 30            // 帧头固定 30 字节
PROTOCOL.FRAME_TRAILER_SIZE   = 4             // 帧尾 CRC32 (uint32 LE)
PROTOCOL.COMMAND_HEADER_SIZE  = 4             // 命令头: opcode(1) + pay_len(3)
PROTOCOL.COMMAND_ALIGNMENT    = 4             // 命令 4 字节对齐
```

## 4. 客户端配置

### 4.1 CONFIG 对象

客户端配置集中在 `client/index.js` 的 `CONFIG` 对象中，通过 `Object.freeze()` 冻结:

```javascript
const CONFIG = Object.freeze({
    /** WebSocket 服务器地址 (生产环境应使用 wss://) */
    SERVER_URL:                'wss://localhost:3000',

    /** DOM 中 Canvas 元素的 id 属性值 */
    CANVAS_ID:                 'main',

    /** 重连初始延迟 (ms) — 指数退避的基准值 */
    RECONNECT_DELAY_MS:        1000,

    /** 重连最大延迟 (ms) — 上限 30 秒 */
    MAX_RECONNECT_DELAY_MS:    30000,

    /** 帧元数据最大保留时间 (3s) — 超时的帧元数据在 pruneFrameMetadata 中清理 */
    FRAME_HISTORY_MAX_AGE_MS:  3000,

    /** 视口 resize 防抖延迟 (ms) — 避免快速拖动窗口时发送大量 viewport 消息 */
    VIEWPORT_RESIZE_DEBOUNCE_MS: 150,

    /** Phase 2 侧信道防御: 在 CanvasKit flush 前注入 ±1ms 随机延迟 */
    JITTER_ENABLED:            true,

    /** 抖动范围 (ms) */
    JITTER_RANGE_MS:           1.0,
});
```

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `SERVER_URL` | `wss://localhost:3000` | WebSocket 服务器 URL，必须以 `wss://` 开头（生产环境） |
| `CANVAS_ID` | `main` | 渲染目标 Canvas 元素的 DOM ID |
| `RECONNECT_DELAY_MS` | 1000 | 重连初始延迟（指数退避基准值，1s 起步） |
| `MAX_RECONNECT_DELAY_MS` | 30000 | 重连最大延迟（指数退避上限，30s） |
| `FRAME_HISTORY_MAX_AGE_MS` | 3000 | 帧元数据保留时间（与 `server/config.js` 一致） |
| `VIEWPORT_RESIZE_DEBOUNCE_MS` | 150 | 视口 resize 防抖延迟 |
| `JITTER_ENABLED` | true | ±1ms 渲染抖动（侧信道防御，Phase 2） |
| `JITTER_RANGE_MS` | 1.0 | 抖动范围 |

### 4.6 客户端协议常量（`client/protocol.js`）

客户端协议常量与服务端、C++ 三端保持严格一致，由 `Object.freeze()` 深度冻结:

```javascript
PROTOCOL.VERSION              = 0x01       // 协议版本号
PROTOCOL.FRAME_HEADER_SIZE    = 30         // 帧头固定 30 字节
PROTOCOL.FRAME_TRAILER_SIZE   = 4          // 帧尾 CRC32
PROTOCOL.COMMAND_HEADER_SIZE  = 4          // 命令头: opcode(1) + pay_len(3)
PROTOCOL.COMMAND_ALIGNMENT    = 4          // 命令 4 字节对齐
```

> 任何修改必须同步更新 `client/protocol.js`、`server/config.js`、`cpp/frame_constants.h` 三处，否则将导致版本不匹配错误。

## 5. 通信协议

### 5.1 通信概览

| 通道 | 协议 | 方向 | 数据格式 |
|------|------|------|---------|
| 帧传输 | WebSocket 二进制 | S → C | 二进制帧（Header + CommandStream + CRC32） |
| 控制消息 | WebSocket 文本 | C ↔ S | JSON 文本消息 |

### 5.2 JSON 控制消息类型

| `type` | 方向 | 字段 | 用途 |
|--------|------|------|------|
| `ready` | C → S | `url` | 通知服务端导航到指定 URL |
| `viewport` | C → S | `width`, `height`, `devicePixelRatio` | 上报视口尺寸（服务端据此调整 Chromium 窗口大小） |
| `io` | C → S | `type`, `x`, `y`, `button`, `buttons`, `deltaX`, `deltaY`, `keyCode`, `key`, `frameId` | HID 输入事件（鼠标/键盘/滚轮） |
| `request_keyframe` | C → S | （无） | 客户端请求关键帧（用于状态重置） |
| `error` | S → C | `code`, `message` | 服务端错误通知 |
| `pong` | S → C | （无） | 心跳响应 |

#### 5.2.1 客户端 → 服务端消息示例

**`ready` — 初始导航**:
```json
{
  "type": "ready",
  "url": "https://example.com"
}
```

**`viewport` — 视口上报**:
```json
{
  "type": "viewport",
  "width": 1920,
  "height": 1080,
  "devicePixelRatio": 1.0
}
```

**`io` — 鼠标移动**:
```json
{
  "type": "io",
  "data": {
    "type": "mousemove",
    "x": 512,
    "y": 384,
    "button": 0,
    "buttons": 0,
    "frameId": 42
  }
}
```

**`io` — 滚轮**:
```json
{
  "type": "io",
  "data": {
    "type": "wheel",
    "deltaX": 0,
    "deltaY": -120,
    "frameId": 42
  }
}
```

**`io` — 键盘按下**:
```json
{
  "type": "io",
  "data": {
    "type": "keydown",
    "keyCode": 65,
    "key": "a",
    "frameId": 42
  }
}
```

**`request_keyframe` — 请求关键帧**:
```json

}
```

### 5.3 二进制帧格式

```
┌─────────────────────────────────────────────────────────────┐
│                       二进制帧 (N 字节)                       │
├─────────────────────────────────────────────────────────────┤
│  [30B Header] [N B CommandStream] [4B CRC32]                │
└─────────────────────────────────────────────────────────────┘

CRC32 覆盖范围: Header(30B) + CommandStream(N B)
CRC32 多项式: 0xEDB88320 (IEEE 802.3 标准，与 zlib/gzip 兼容)
```

#### 5.3.1 帧头结构（30 字节）

| 偏移 | 大小 | 字段 | 说明 |
|------|------|------|------|
| 0 | 1B | version | 协议版本 (0x01) |
| 1 | 1B | flags | 标志位（bit0: keyframe, bit1: hasFontData, bit2: hasDirtyRects） |
| 2 | 4B | frame_id | 帧 ID（uint32 LE，单调递增） |
| 6 | 8B | timestamp | 帧时间戳（int64 LE，毫秒） |
| 14 | 4B | scroll_x | 水平滚动偏移（int32 LE） |
| 18 | 4B | scroll_y | 垂直滚动偏移（int32 LE） |
| 22 | 2B | viewport_w | 视口宽度（uint16 LE，CSS 像素） |
| 24 | 2B | viewport_h | 视口高度（uint16 LE，CSS 像素） |
| 26 | 2B | canvas_w | Canvas 宽度（uint16 LE） |
| 28 | 2B | canvas_h | Canvas 高度（uint16 LE） |

### 5.4 命令格式

```
┌─────────────────────────────────────────────────────────────┐
│                       单条命令 (4 字节对齐)                  │
├─────────────────────────────────────────────────────────────┤
│  [1B Opcode] [3B PayloadLength (uint24 LE)] [N B Payload]   │
│                       [0~3B Padding (全零)]                 │
└─────────────────────────────────────────────────────────────┘

每条命令 = COMMAND_HEADER_SIZE(4B) + Payload(N B) + 4 字节对齐填充
Padding 字节必须全零，由 CommandValidator 第 7 层校验
```

### 5.5 Opcode 分配

| 范围 | 类别 | 说明 |
|------|------|------|
| `0x00` | 保留 | 空操作标记 |
| `0x01-0x0F` | 状态管理 | save, restore, saveLayer |
| `0x10-0x1F` | 变换 | translate, scale, rotate, concat, concat44 |
| `0x20-0x2F` | 裁剪 | clipRect, clipRRect, clipPath |
| `0x30-0x3F` | 形状绘制 | rect, rrect, oval, arc, path, points, region |
| `0x40-0x4F` | 图像绘制 | image, imageRect, lattice, atlas, patch, edgeAA |
| `0x50-0x5F` | 文本绘制 | textBlob, glyphRunList |
| `0x60-0x6F` | 其他绘制 | paint, color, shadow, vertices, drawable, annotation |
| `0x70-0x7F` | 扩展 | fontData, imageData, setMatrix, drawScrollbar, noop |
| `0x80-0xFF` | **非法** | 客户端必须拒收 |

详见 [`protocol/opcodes.md`](../protocol/opcodes.md)。

## 6. 安全配置

### 6.1 TLS 配置

生产环境**必须**启用 TLS。配置通过环境变量完成:

```bash
export WISON_TLS_ENABLED=1
export WISON_TLS_CERT=/etc/ssl/certs/wison-rbi-fullchain.pem
export WISON_TLS_KEY=/etc/ssl/private/wison-rbi-privkey.pem
export WISON_TLS_CA=/etc/ssl/certs/client-ca.pem   # 可选: mTLS
```

| 配置项 | 值 | 说明 |
|--------|------|------|
| 最低版本 | TLSv1.3 | 强制 TLS 1.3，禁用所有旧版本 |
| 密码套件 | `TLS_AES_256_GCM_SHA384` / `TLS_CHACHA20_POLY1305_SHA256` / `TLS_AES_128_GCM_SHA256` | 仅允许 AEAD 套件（明确禁用 CBC/RC4/3DES） |
| HonorCipherOrder | true | 服务端决定密码套件优先级（防降级攻击） |
| mTLS | 可选 | 通过 `WISON_TLS_CA` 启用客户端证书验证 |

#### 6.1.1 证书获取

**Let's Encrypt（推荐）**:

```bash
# 私钥:   /etc/letsencrypt/live/wison-rbi.example.com/privkey.pem
```

**自签名证书（仅测试用）**:

#### 6.2.3 沙箱禁用变量警告

| 变量 | 风险等级 | 允许场景 |

### 6.3 安全审计要点

以下审计点均已对照代码实现验证:

| # | 审计项 | 实现状态 | 验证位置 |
|---|--------|---------|---------|
| 1 | 客户端使用原生 WebSocket API（无 socket.io-client 依赖） | ✅ 已实现 | `client/index.js` 第 264 行 `new WebSocket(CONFIG.SERVER_URL)` |

#### 6.3.1 字体 Magic 白名单详情

仅允许 SFNT 系列和 WOFF2 格式。**WOFF v1 (`0x774F4646`) 明确拒绝**:

| Magic 值 | 说明 |
|----------|------|
| `0x00010000` | TrueType（经典 sfVersion） |
| `0x4F54544F` | 'OTTO' — OpenType CFF |
| `0x74727565` | 'true' — TrueType (Apple) |
| `0x74746366` | 'ttcf' — TrueType Collection |
| `0x774F4632` | 'wOF2' — WOFF2 |

校验失败 → 替换为 CanvasKit 回退字体 (`font_id=0`)。

### 6.4 客户端命令校验（9 层纵深防御）

| 层级 | 校验内容 |
|------|---------|
| 1 | Opcode 白名单: 仅允许 0x01-0x7F |

### 6.5 威胁模型与对策

| 威胁 | 对策 | 实现位置 |
|------|------|---------|

### 8.1 反向代理（nginx）

生产环境推荐在 Wison-RBI 服务前部署 nginx，承担 TLS 终止与负载均衡:

```nginx
upstream wison_rbi_backend {
    least_conn;
    server 10.0.0.1:3000 max_fails=3 fail_timeout=30s;
    server 10.0.0.2:3000 max_fails=3 fail_timeout=30s;
    server 10.0.0.3:3000 max_fails=3 fail_timeout=30s;
    server 10.0.0.4:3000 max_fails=3 fail_timeout=30s;
}

server {
    listen 443 ssl http2;
    server_name wison-rbi.example.com;

    # TLS 终止（卸载后端 TLS 负担）
    ssl_certificate     /etc/ssl/certs/wison-rbi-fullchain.pem;
    ssl_certificate_key /etc/ssl/private/wison-rbi-privkey.pem;
    ssl_protocols       TLSv1.3;
    ssl_ciphers         TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256:TLS_AES_128_GCM_SHA256;

    # WebSocket 长连接
    location / {
        proxy_pass http://wison_rbi_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # WebSocket 长连接超时（默认 60s 太短）
        proxy_read_timeout 86400s;
        proxy_send_timeout 86400s;
    }

    # 健康检查端点（短连接即可）
    location /health {
        proxy_pass http://wison_rbi_backend;
        access_log off;
    }
}
```

### 8.5 自动伸缩

基于 `sessions_active` 指标进行水平自动伸缩:

```yaml
# Kubernetes HPA
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: wison-rbi-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: wison-rbi

  maxReplicas: 10
```

**伸缩策略**:
- **扩容**: `sessions_active` 平均值 > 3（每实例 4 个会话上限，预留 25% 余量）
- **缩容**: `sessions_active` 平均值 < 1，持续 5 分钟
- **冷却**: 扩容 60s，缩容 300s（避免抖动）

## 9. 配置变更流程

### 9.1 变更工作流

```
1. 修改配置源码
   ├─ 服务端: server/config.js
   └─ 客户端: client/index.js (CONFIG 对象) 或 client/protocol.js

2. 应用变更
```

## 10. 常见问题

### 10.1 客户端连接问题

**Q1: 客户端连接立即断开，Console 显示 `ws_error`**

可能原因:
- TLS 证书无效或客户端不信任 CA
- 服务端未启动或端口被防火墙拦截
- `SERVER_URL` 协议错误（应使用 `wss://`，非 `ws://`）

排查:
```bash
# 服务端: 验证 TLS 证书
openssl s_client -connect localhost:3000 -showcerts

# 服务端: 检查端口监听
ss -tlnp | grep 3000

# 客户端: 检查 Console 详细错误
```

**Q2: 客户端白屏，CanvasKit 加载失败**

可能原因:
- CanvasKit WASM 文件路径错误或 MIME 类型错误
- 网络加载超时

排查:
- 检查 `client/` 目录下 `canvaskit.wasm` 是否存在
- 检查扩展的 `web_accessible_resources` 配置是否包含 wasm 文件
- 检查 Console 是否有 `canvaskit_loaded` 日志

### 10.2 渲染问题

**Q3: 画面不更新，服务端 `frames_dropped` 计数增加**

可能原因:
- 帧大小超过 `MAX_COMPRESSED_FRAME` (4 MB)
- 网络带宽不足
- 帧被 CRC 校验失败拒绝

排查:
```bash
# 检查服务端日志
journalctl -u wison-rbi -f | grep -E 'frame_drop|crc_fail'

# 检查指标
curl http://localhost:3000/metrics | jq '.counters.frames_dropped'
```

**Q4: 增量帧异常，画面错乱**

可能原因:
- `frame_id` 不连续，触发 `request_keyframe` 但未及时响应
- 客户端帧历史过期（超过 `FRAME_HISTORY_MAX_AGE_MS` = 3s）导致坐标转换错误

排查:
- 检查服务端是否频繁重启（`frame_id` 归零）
- 检查网络延迟是否过高（应 <100ms RTT）

**Q5: 字体显示错乱**

可能原因:
- 字体 Magic 校验失败（非 SFNT/WOFF2 格式）
- 字体文件超过 `MAX_FONT_INLINE_BYTES` (5 MB)

排查:
- 检查服务端 `font_validator.js` 日志
- 客户端 Console 检查 `font_magic_reject` 事件

### 10.3 性能问题

**Q6: Chromium 频繁重启，OOM**

可能原因:
- 单实例内存超过 `MAX_MEMORY_BYTES` (2 GB)
- 服务器总内存不足

解决:
- 增加服务器内存
- 降低 `MAX_SESSIONS`
- 检查页面是否有内存泄漏

**Q7: 客户端 CPU 占用过高**

可能原因:
- 帧率超过 60fps
- `JITTER_ENABLED` 导致额外调度开销

解决:
- 确认 `requestAnimationFrame` 调度正常
- 临时禁用 `JITTER_ENABLED` 排查（仅测试环境）

### 10.4 配置问题

**Q8: 修改 `client/index.js` 后无效果**

原因: Chrome 扩展未重新加载。

解决:
1. 打开 `chrome://extensions`
2. 找到 Wison-RBI 扩展
3. 点击"刷新"按钮（圆形箭头）
4. 关闭并重新打开扩展入口页

**Q9: 修改 `server/config.js` 后无效果**

原因: 服务端未重启。

解决:
```bash
sudo systemctl restart wison-rbi
sudo journalctl -u wison-rbi -f  # 验证启动成功
```

**Q10: `GARNET_UNSAFE_NO_SANDBOX` 在生产环境意外启用**

风险: 极高，宿主机直接暴露于 Chromium 漏洞。

排查:
```bash
# 检查 systemd 单元文件
systemctl cat wison-rbi | grep GARNET_UNSAFE_NO_SANDBOX

# 检查进程环境
cat /proc/$(pgrep -f "node server.js")/environ | tr '\0' '\n' | grep GARNET
```

解决: 立即从 systemd 单元文件中删除该环境变量并重启服务。

## 11. 性能指标

### 11.1 Phase 1 性能目标

| 指标 | Phase 1 目标 | 实现策略 |
|------|-------------|---------|
| 端到端延迟 | <150 ms | 异步 CDP + 流水线化 |
| 帧生成延迟 | <5 ms | Compositor 线程录制 + Worker 线程编码 |
| 首帧带宽 | <3 MB (gzip) | 原生 zlib + hash-ref 去重 |
| 客户端 CPU | <30% @60 fps | `requestAnimationFrame` 调度 |
| 客户端内存 | <256 MB | 字体 LRU (64 MB) + 图像 LRU (64 MB) |

### 11.2 性能调优建议

#### 11.2.1 并发优化

```bash
# 增加 Chromium 实例数（需配合服务器内存）
export MAX_SESSIONS=8

# 调整 Node.js 内存上限
node --max-old-space-size=4096 server.js
```

#### 11.2.2 网络优化

```bash
# 启用 TCP BBR 拥塞控制
sudo sysctl -w net.ipv4.tcp_congestion_control=bbr

# 调整 somaxconn
sudo sysctl -w net.core.somaxconn=4096
```

> **注意**: 帧已在应用层 gzip 压缩，**不要**额外启用 WebSocket permessage-deflate，否则会导致双重压缩开销。

#### 11.2.3 Chromium 优化

```bash
# 减少 Chromium 内存占用
chromium --max_old_space_size=2048

# 禁用不必要的功能
chromium --disable-background-networking --disable-sync --disable-default-apps
```

### 11.3 性能基准测试

```bash
cd server
node benchmark.js --iterations=500
```

预期结果（参考）:
- 帧组装: <2 ms (medium 帧)
- gzip 压缩: 压缩比 10-50%
- 首帧带宽: <1 MB (1 MB raw → gzip)
- 增量帧带宽: <50 KB (1 KB raw → gzip)
- CRC32: >500 MB/s
